## 2. Bug corregido: descarga de vídeo anotado (PermissionError)

### Síntoma
Al pulsar "Descargar vídeo anotado" en la app, aparecía error 500.

### Causa raíz
`app.py` usaba `send_file()` de Flask para enviar el archivo generado, y en el bloque `finally` hacía `os.remove()`. En **Windows**, `send_file()` mantiene un handle abierto sobre el archivo mientras lo transmite, y al intentar borrarlo simultáneamente se produce:

```
PermissionError: [WinError 32] El proceso no tiene acceso al archivo
porque está siendo utilizado por otro proceso
```

### Solución aplicada
**Archivo:** `modules/salto/backend/app.py`

**Antes:**
```python
return send_file(
    ruta_salida,
    mimetype="video/mp4",
    as_attachment=True,
    download_name="salto_anotado.mp4",
)
# ...
finally:
    for ruta in [ruta_entrada, ruta_salida]:
        if os.path.exists(ruta):
            os.remove(ruta)
```

**Después:**
```python
# Leer a memoria antes de eliminar para evitar PermissionError en Windows
with open(ruta_salida, "rb") as f:
    video_bytes = f.read()

return Response(
    video_bytes,
    mimetype="video/mp4",
    headers={
        "Content-Disposition": 'attachment; filename="salto_anotado.mp4"',
        "Content-Length": str(len(video_bytes)),
    },
)
# ...
finally:
    for ruta in [ruta_entrada, ruta_salida]:
        try:
            if os.path.exists(ruta):
                os.remove(ruta)
        except OSError:
            pass
```

También se añadió `Response` al import de Flask:
```python
from flask import Flask, Response, jsonify, request, send_file
```

## 3. Hallazgo crítico: procesos zombie en puerto 5001

### Síntoma
La API devolvía `0.0 cm` para vídeos que el script directo procesaba correctamente (p.ej. `Horizontal prueba.mp4` → 330 cm directo vs 0 cm vía API).

### Causa raíz
Había **dos procesos Python** escuchando en el puerto 5001:
- **PID 15176** — proceso viejo del 9 de abril a las 22:30 (código desactualizado)
- **PID 2616** — proceso nuevo del 10 de abril a las 9:44

El proceso viejo interceptaba las peticiones HTTP y respondía con código antiguo que no tenía las correcciones de slow-motion ni el peak detector, devolviendo siempre 0 cm.

### Solución
```powershell
Stop-Process -Id 2616,15176 -Force
```

### Lección
Siempre verificar con `netstat -ano | Select-String ":5001 " | Select-String "LISTEN"` que no haya procesos zombie antes de arrancar el backend.

## 5. Diagnóstico: datos vacíos en "Biomecánica del aterrizaje"

### Síntoma
En el panel de aterrizaje, 3 campos aparecen como `--`:
- Oscilación CM → `-- px`
- T. Estabilización → `-- s`
- Estable → `--`

Mientras que otros SÍ se muestran:
- Ang. Rodilla Aterr. → `155.41 deg`
- Flex. Máxima → `155.41 deg`
- Amortiguación → `0 deg`
- Asim. Recepción → `52%`

### Causa raíz
`AterrizajeService.analizar_estabilidad()` requiere **mínimo 3 frames post-aterrizaje** para calcular la oscilación del centro de masa. En el vídeo `Horizontal prueba.mp4`:
- Aterrizaje = frame 120
- Total frames = 121 (índices 0–120)
- Frames post-aterrizaje = **1 solo frame** → la función devuelve `None`

### Por qué amortiguación = 0°
El ángulo de rodilla al aterrizar (155.41°) es igual a la flexión máxima post-aterrizaje (155.41°), por lo que el rango de amortiguación = 0°. Esto se debe a que el vídeo es **generado por IA** y la biomecánica no es realista (la rodilla no se flexiona al aterrizar como en un salto real).

### No es un bug
Con un vídeo real donde la cámara grabe al menos 1-2 segundos después de aterrizar, todos estos campos se llenarán correctamente.

## 7. Cámara en tiempo real desde móvil (HTTPS)

### Síntoma
Al abrir la app desde el móvil (Chrome, `http://192.168.1.17:8080`) e intentar usar la cámara, el navegador bloquea el acceso con el mensaje de que no se puede usar la cámara.

### Causa raíz
La API `navigator.mediaDevices.getUserMedia()` requiere un **contexto seguro** (Secure Context). Los navegadores solo lo permiten en:
- `https://` (cualquier dominio)
- `http://localhost` (excepción especial)

Al acceder desde el móvil vía `http://192.168.1.17:8080`, no es localhost ni HTTPS → el navegador bloquea la cámara.

### Solución aplicada

#### 1. Certificado SSL autofirmado
**Archivo nuevo:** `scripts/generate_cert.py`

Genera un par de archivos en `certs/`:
- `cert.pem` — certificado público (válido 365 días)
- `key.pem` — clave privada RSA 2048 bits

El certificado incluye como Subject Alternative Names:
- `localhost`
- `127.0.0.1`
- IP LAN detectada automáticamente (e.g. `192.168.1.17`)

```bash
python scripts/generate_cert.py
```

#### 2. Servidor HTTPS frontend
**Archivo nuevo:** `scripts/https_server.py`

Servidor HTTP simple de Python envuelto con SSL. Sirve `integration/web/` en **puerto 8443** con el certificado generado.

```bash
python scripts/https_server.py
# → https://localhost:8443
# → https://192.168.1.17:8443
```

#### 3. Backend Flask con SSL automático
**Archivo modificado:** `modules/salto/backend/app.py`

Ahora al arrancar, Flask busca `certs/cert.pem` y `certs/key.pem` en la raíz del proyecto. Si los encuentra, arranca con HTTPS automáticamente:

```python
_ssl_context = (_cert, _key) if os.path.exists(_cert) and os.path.exists(_key) else None
app.run(host="0.0.0.0", port=FLASK_PORT, debug=False, ssl_context=_ssl_context)
```

Resultado:
```
 * Running on https://127.0.0.1:5001
 * Running on https://192.168.1.17:5001
```

#### 4. Detección de protocolo en frontend
**Archivo modificado:** `integration/web/js/config.js`

La función `getBackendBaseUrl()` detecta el protocolo actual de la página para generar URLs coherentes (evita errores de mixed content HTTPS→HTTP):

```javascript
function getBackendBaseUrl() {
    const protocol = window.location.protocol === 'https:' ? 'https' : 'http';
    return `${protocol}://${getCurrentHost()}:${BACKEND_SALTO_PORT}`;
}
```

#### 5. Seguridad: claves excluidas del repo
**Archivo modificado:** `.gitignore`

Añadida línea `certs/` para que `cert.pem` y `key.pem` no se suban al repositorio.

### Uso desde el móvil

1. Arrancar ambos servidores:
   ```bash
   python scripts/https_server.py                    # frontend :8443
   python modules/salto/backend/app.py               # backend  :5001
   ```
2. En Chrome del móvil, ir a `https://192.168.1.17:8443`
3. El navegador mostrará advertencia de certificado (es autofirmado → **normal**)
4. Pulsar **"Avanzado" → "Continuar de todos modos"**
5. También aceptar el certificado del backend visitando `https://192.168.1.17:5001` y aceptando ahí también
6. Volver a `https://192.168.1.17:8443` — la cámara debería funcionar

> **Nota:** Si cambia la IP de la máquina, regenerar el certificado con `python scripts/generate_cert.py`.

## 8. Mejora de precisión de landmarks en tiempo real

### Síntoma
Los landmarks del esqueleto dibujados sobre la cámara en vivo temblaban mucho y no se ajustaban bien al cuerpo.

### Causa raíz
- Se usaba el modelo **lite** de MediaPipe, menos preciso.
- No había umbrales de confianza explícitos (valores por defecto ~0.5).
- No se aplicaba ningún suavizado a los landmarks entre frames.

### Solución aplicada
**Archivo modificado:** `integration/web/js/api_salto.js`

#### 1. Modelo `full` en vez de `lite`
```javascript
// Antes
modelAssetPath: '.../pose_landmarker_lite/float16/1/pose_landmarker_lite.task'

// Después
modelAssetPath: '.../pose_landmarker_full/float16/1/pose_landmarker_full.task'
```
El modelo `full` (~6MB vs ~4MB) es significativamente más preciso, especialmente en extremidades. Rendimiento aceptable en móviles modernos con GPU delegate.

#### 2. Umbrales de confianza subidos a 0.6
```javascript
minPoseDetectionConfidence: 0.6,
minTrackingConfidence: 0.6
```
(Antes: valores por defecto ~0.5). Descarta detecciones ruidosas.

#### 3. Filtro EMA (Exponential Moving Average)
```javascript
const EMA_ALPHA = 0.4;  // 0 = muy suave, 1 = sin filtro
```
Suaviza la posición de los 33 landmarks frame a frame, eliminando el temblor sin lag perceptible. Se resetea al iniciar cada sesión de detección.

## 9. Bug corregido: observaciones contradictorias en tiempo real

### Síntoma
Al usar la cámara en vivo para un salto horizontal, la app mostraba 103cm como distancia pero más abajo aparecía "No se detectó un salto válido" en las observaciones automáticas.

### Causa raíz
Cuando la cámara detecta un salto en vivo:
1. MediaRecorder graba un WebM simultáneamente.
2. Si `videoBlob` existe, se envía al backend vía `/api/salto/calcular`.
3. El backend reprocesa el WebM con MediaPipe → a menudo obtiene 0cm (formato WebM de MediaRecorder no es ideal).
4. El backend devuelve `clasificacion: "no_detectado"` + `observaciones: ["No se detectó un salto válido..."]`.
5. El código de fallback hacía `{...payload, ...fallbackLocal}` → `fallbackLocal` restauraba la distancia local (103cm) pero **no incluía** campos `observaciones`/`alertas`/`clasificacion`, así que los del backend (que decían "no válido") sobrevivían al merge.

**Resultado:** 103cm + "No se detectó salto válido" simultáneamente.

### Solución aplicada
**Archivo modificado:** `integration/web/js/api_salto.js`

En el path de fallback (backend=0cm, local>0cm), se sobreescriben explícitamente los campos de interpretación:
```javascript
return {
    ...payload,
    ...fallbackLocal,
    // ...métricas...
    observaciones: ["Salto medido en tiempo real. El vídeo grabado no pudo ser reprocesado por el backend."],
    alertas: [],
    clasificacion: 'medido_en_vivo',
};
```

En el path sin vídeo (`guardarLocalEnSaltos`), también se incluyen estos campos:
```javascript
return {
    ...datosLocales,
    id_salto: payloadLocal.id_salto,
    datos_parciales: true,
    observaciones: ["Salto medido en tiempo real."],
    alertas: [],
    clasificacion: 'medido_en_vivo',
};
```

## 10. Mejora de fiabilidad de métricas en tiempo real

### Síntoma
Las métricas biomecánicas en modo cámara en vivo (confianza, estabilidad, ángulos, asimetría) eran siempre casi idénticas entre saltos diferentes.

### Causa raíz

| Métrica | Problema |
|---------|----------|
| Confianza | Hardcoded `0.95` — siempre idéntica |
| Estabilidad | Fórmula con confianza fija → resultado siempre ~84-98 |
| Ángulos | Solo se capturaban en 1 frame (el de despegue) |
| Asimetría | Solo se medía en despegue, no en aterrizaje |

### Solución aplicada
**Archivo modificado:** `integration/web/js/api_salto.js`

#### Variables de tracking añadidas
```javascript
let angulosAterrizajeActual = { angulo_rodilla_deg: null, angulo_cadera_deg: null };
let asimetriaAterrizajeActual = null;
let minRodillaDuranteVuelo = null;
let minCaderaDuranteVuelo = null;
let visibilidadAcumulada = 0;
let framesConVisibilidad = 0;
let yPostAterrizaje = [];
```

#### 1. Confianza real (ya no hardcoded)
```javascript
function calcularConfianzaDesdeVisibilidad(landmarks) {
    const indices = [23, 24, 25, 26, 27, 28, 29, 30]; // caderas, rodillas, tobillos, talones
    // Media de visibility real de MediaPipe → varía con iluminación, ángulo, oclusión
}
```
Se acumula en cada frame y se promedia al finalizar.

#### 2. Estabilidad real post-aterrizaje
```javascript
function calcularEstabilidadPostAterrizaje(muestrasY) {
    // Desviación estándar de posición Y de pies durante 500ms post-aterrizaje
    // Menor oscilación = mayor puntuación (0-100)
}
```
Nuevo estado `'aterrizaje_reciente'` en la máquina de estados que recoge ~15 frames (500ms a 30fps) de muestras post-aterrizaje antes de finalizar el salto.

#### 3. Ángulos en 3 momentos (despegue + vuelo + aterrizaje)
La máquina de estados ahora trackea:
- **Despegue:** ángulos de rodilla y cadera al iniciar el salto.
- **Vuelo:** flexión máxima de rodilla y cadera durante todo el vuelo.
- **Aterrizaje:** ángulos y asimetría al momento de recepción.

#### 4. Asimetría en aterrizaje
Se mide la diferencia de altura Y entre pie izquierdo y derecho en el frame de aterrizaje (más relevante biomecánicamente que en despegue).

### Evaluación de fiabilidad post-cambio

| Métrica | Fiabilidad | Notas |
|---------|-----------|-------|
| Tiempo de vuelo | **Alta** | `performance.now()` directo |
| Confianza | **Alta** | Media real de `visibility` de 8 landmarks |
| Distancia | **Media-Alta** | Vertical: 60% visual + 40% cinemática. Horizontal: escala calibrada |
| Potencia | **Media** | Sayers (vertical) o cinética (horizontal), validadas en literatura |
| Ángulos | **Media** | 3 momentos reales, pero pocos frames de vuelo |
| Estabilidad | **Media** | Desviación estándar real de ~15 muestras post-aterrizaje |
| Asimetría | **Media** | Frame único de aterrizaje |

**Fiabilidad general estimada: ~70-80%** vs. backend con vídeo (~90%). La diferencia principal es que el backend analiza todos los frames del vídeo, mientras que en vivo se trabaja con momentos clave.
